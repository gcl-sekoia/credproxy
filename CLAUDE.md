# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository status

The product (codename "credproxy") is a transparent egress proxy for workspace containers — LLM-agent sandboxes, CI runners, dev shells, batch jobs. The scheme-aware credential-injection model — the typed-scheme model that replaced the original substring-replace — is **fully implemented**: substitute family (`bearer`/`basic`/`body`), sign family (`sigv4`), and re-seal family (`oauth2-reseal` — mints the token-endpoint response into a TTL'd dynamic placeholder so even the short-lived token stays out of the workspace). The sandboxed Starlark escape hatch (`proxy/starlark_runtime.py`, via the `starlark-pyo3` proxy dependency) is wired through the scripted-injector authoring contract — an injector with `scheme = "script"` declares `family`/`slots`/`location` (+ an `api` version) in TOML and names a `.star` file (builtin examples in `cli/credproxy_cli/builtin/scripts/`); the CLI pushes the script source, the proxy compiles it into a `ScriptedScheme`. The script primitive API is the flat/implicit-ctx "option B" model: zero-arg `on_request()`/`on_response()` hooks, flat `req_*`/`resp_*` primitives reading a contextvar-bound ctx (no ctx threading), carrier crypto (so SigV4-class multi-round signing is expressible), `jwt_encode_sign`/`json_*`, and `mint`/`mint_into_json` for re-seal; builtin `ovh`/`jwt-bearer`/`oauth-reseal` worked examples ship and are tested. All are useful background; divergence from earlier docs is expected (learn-by-building). CLAUDE.md (this file) and the code are the living source of truth. **Pre-release — there are no users, so there is no backward-compatibility or migration burden: optimize purely for the best end-state design.** Rename, remove, or restructure freely (config formats, CLI flags, file layouts, on-disk state) without compat shims, deprecation aliases, or churn worry — just update the call sites, tests, and docs to match. Never keep a worse design to avoid churn. The repo holds the proxy core under `proxy/`, the host CLI at `bin/credproxy` and `bin/credp`, the CLI package under `cli/credproxy_cli/`, tests under `tests/`, and `docs/`.

## Big-picture architecture

The product is **two containers that must stay separated**:

1. **Proxy container** (Linux, requires `NET_ADMIN`): owns the netns, installs iptables rules, runs two listeners — mitmproxy on `127.0.0.1:39999` (transparent intercept) and a single aiohttp HTTP API on `0.0.0.0:39998` that serves both workspace-facing bootstrap routes and host-facing admin routes. iptables redirects sentinel-IP `:80` to the HTTP listener and everything-else-TCP to mitmproxy. The HTTP listener is port-published to the host as `127.0.0.1:<ephemeral port>` (Docker assigns a free port per proxy container so multiple workspaces run simultaneously without conflict); workspace reaches it through the sentinel redirect or directly via `127.0.0.1:39998` in the shared netns.
2. **Host CLI** (`bin/credproxy`, Python; Go later): a workspace manager. A credproxy *workspace* is a named, persistent pair of containers — a proxy container and a workspace container sharing one netns — created once, then started/stopped explicitly; multiple workspaces run concurrently. Config is TOML, one file per workspace at `$XDG_CONFIG_HOME/credproxy/workspaces/<name>.toml`. State lives at `$XDG_STATE_HOME/credproxy/workspaces/<name>/`. The host port is ephemeral and resolved at call time via `docker port`; it is never persisted. Mount-target paths are sourced from the proxy image's `ENV` declarations (read via `docker inspect`'s `Config.Env`), so the image is the single source of truth for its own API; the CLI hardcodes only its own conventions (image tag, workspace storage layout, container/volume naming, default workspace image).

The workspace container is **the user's** image — never modified, never granted privilege. This "bring your own image" constraint is load-bearing. See `docs/reference/workspace.md` for the constraints joining the proxy's netns imposes.

**SELinux (Fedora/RHEL hosts):** host bind mounts otherwise get denied under enforcing SELinux. Split by trust: the proxy stays confined and relabels only its own mounts (token `:Z`, dev-source `:z`); the workspace runs with `--security-opt label=disable` (distrobox/toolbx style) so the user's bind-mounted project dirs work **without** relabeling/mutating them. No-op on non-SELinux hosts. Validated on Fedora rootless podman. See `docs/reference/workspace.md`.

Traffic flow: workspace egress → iptables OUTPUT in shared netns → REDIRECT to mitmproxy (or to the HTTP API for sentinel:80) → SNI peek → either substitute-placeholder-and-forward (terminate TLS) or passthrough (`client_hello.ignore_connection = True`).

**Configuration flow**: each workspace has its own bearer token at `$XDG_STATE_HOME/credproxy/workspaces/<name>/auth.token` (mode 0644), generated by `credproxy workspace create` or `start` if absent and bind-mounted read-only into that workspace's proxy at `/run/secrets-ro/auth.token`. The proxy reads it fresh on every `/admin/config` request — no staging copy, no in-memory snapshot — so rotating the host file takes effect on the next request without a proxy restart. Config lives on tmpfs at `/run/secrets/config.json`, written by `POST /admin/config`. Lifecycle: the token is host-owned and survives everything; the tmpfs config does **not** survive a `docker stop`/`start`, so `credproxy workspace NAME start` always re-materializes and re-pushes bindings once the proxy is healthy. `credproxy dev reload` (SIGHUP) re-execs python in place and the tmpfs survives that, so pushed config persists across a reload. **Two readiness signals, never collapsed** (`proxy/bootstrap.py`): `/health` is capture-ready (intercept listener accepting + CA on disk, creds-agnostic), `/ready` additionally requires `config_generation ≥ 1` — a monotonic counter `POST /admin/config` bumps on every accepted push (persisted in the tmpfs envelope, so it survives a reload but resets to 0 on stop/start, correctly matching the re-push). A **rules-only** config goes ready like any other (gating is on the counter, not non-empty bindings). `--wait`/`start` poll `/health`, **never** `/ready` (that gates on the very push they precede — deadlock; #41 I1/I2).

**Attached workspaces** (`docs/advanced/composability.md`): a workspace may declare an `attach` table (`compose_project`/`container`/`discover`/`admin_url`, exactly one; `compose_project` is sugar for the Compose labels) **instead of** container fields (`load_config` enforces mutual exclusion), for when the proxy + workspace containers are run by an external manager (Compose/devcontainers/CI) and credproxy owns only the credentials. It is a first-class workspace — same name/token/state/drift/cwd-addressing. `push` (resolve secrets + POST the full wire config to the attach target, port resolved per call, I6) is the extracted step-(2) engine `start`/`apply` already call; `apply`≡`push` on attached, and `start`/`stop`/`recreate`/`enter`/`exec`/`logs`/`mount`/`dev reload` **refuse** (containers are external — point at `push`). `resolve` builds the wire config without a proxy; the stateless `credproxy push --admin/--config/--token` is the no-workspace CI escape hatch; `emit-compose` scaffolds the sidecar. The shared resolve+POST engine is `core/push.py`.

**Bindings**: a *binding* ties an injector (how a credential is shaped into a request — which typed *scheme* the proxy runs) to a provider (where the value comes from), scoped to one or more hosts. Each host is a literal (exact match, the fast dict path) or a glob pattern containing `*` (matched via `proxy/hostmatch.py`; `*` spans dots, so `*.amazonaws.com` covers every AWS regional endpoint with one binding — the sigv4 scheme reads region/service per-request). Patterns are validated strictly (two rightmost labels must be literal: `*.example.com` ok, `*.com`/`*` rejected); the CLI mirrors that validation in `core/hostmatch.py` so a bad pattern fails at `binding add`. The host CLI resolves each binding's secret(s) from its provider at `start`/`apply`/`binding test` time and pushes the resolved wire config (`{"bindings": [{name, hosts, scheme, params, secret:{slot:value}, placeholder, env}]}`) to `/admin/config`; the proxy dispatches on `scheme` (substitute `bearer`/`basic`/`body`, sign `sigv4`, re-seal `oauth2-reseal`, or `script` for a sandboxed Starlark injector). Re-seal additionally mints the token-endpoint response into a TTL'd runtime substitution layer (dynamic placeholders), so the swap set is `static pushed config + runtime-augmentable layer`, never baked immutable at push time. The provider exec protocol is batch-native (`{op:"get", secrets:[refs]}` → `{values:{ref:val}}`), and the CLI batches across bindings: at resolve time (`start`/`apply`/`binding test`) it groups every binding's refs by provider and invokes each provider **once** with the deduped union, so a provider with a costly setup (a vault that must unlock, a session that must authenticate) pays that cost once per resolve rather than once per binding (`resolve_secrets` in `core/bindings.py`). The real secret never enters the workspace. The inward `/setup` endpoint exposes only least-disclosure fields (name, placeholder, env, scheme, params, hosts — never secret values/provider/secret-id) so the workspace can self-configure. The scheme model lives in the code (`proxy/schemes.py`, `proxy/starlark_runtime.py`) and `docs/reference/injectors.md`.

**Rules**: a *rule* is the credential-free sibling of a binding — it governs traffic on an intercepted host (block a request, answer with a stub, rewrite headers, or run a sandboxed Starlark script) and holds no secret/provider/placeholder. Rules ride the same wire config as a `rules` array next to `bindings`, and a `rule` CLI noun mirrors `binding` exactly (`add`/`remove`/`list`/`test`, loose aliases, `RESERVED_NAMES`). **Pipeline order is a security invariant, not a preference** (`proxy/addon.py`): request rules run **before** injection (a blocked request never receives a credential; a rewrite happens before sigv4 signs), response rules run **after** re-seal (a token-endpoint response is already sealed into a placeholder). Consequence: rule code — declarative or script — **never observes a real credential** (inert placeholders on the request side, exactly what the workspace would see on the response side), so rules add no exfiltration surface — do not reorder this. Evaluation is strict declaration order: rewrites are cumulative, the first terminal action (`block`/`respond`, or a `script` that calls one) short-circuits. A rule script that errors **fails closed toward the policy** (`502 credproxy: rule 'NAME' failed`) — distinct from injector scripts, which fail closed to protect the *secret* (logged by exception type + a safe `source:line` location, extracted from the credproxy-chosen filename only — **never** the message, which could be `fail(secret())`; #33 rung 3). The **interception set is a UNION** of binding hosts and rule hosts (`Credentials.intercepts`), so a rule-only host is TLS-terminated — a documented side effect: adding a rule to a previously-passthrough host flips it to intercepted (CA-cert error, not a 403, if the workspace hasn't bootstrapped). A per-rule **`visible` flag** bundles enumeration (in `/setup`) + attribution (`X-Credproxy-Rule` header + `{"credproxy":{"blocked_by":…}}` body); terminal actions default visible, rewrite/script default hidden. Hidden means hidden from the *workspace*, never the *operator* (logs/audit/`rule test`/`rule list` always show it). Rule scripts use `starlark_runtime.py`'s `kind="rule"` profile — a restricted primitive set (no `secret`/`mint`/crypto, plus `block`/`respond` sinks, **plus** `param()` — the same config-read primitive injectors have) enforced at compile, and **unsanitized** error messages (a rule holds no secret, so full tracebacks are safe — an authoring-UX win). A `script`-action rule may carry a `[rule.params]` table (JSON-clean, script-only) so one **shared** registry script is configured per-rule instead of forked — `param(key, default)` reads it (defaults in the script → zero-config still runs); params are bound **per-rule at eval time** (two rules can share one compiled script), copied at the Starlark boundary (no cross-request/rule mutation). Params are **config, not secrets**: operator-plaintext in TOML/`rule list`/`applied-rules.json`, **excluded from `/setup`** even when `visible` (mirror validators in `core/rules.py`↔`proxy/config.py`; wire-parity + `/setup`-exclusion tested; #35). Rules are NOT hostname egress allowlisting (a non-goal): they govern HTTP on hosts you *name*; unnamed hosts stay passthrough and there is no deny-by-default (hostmatch rejects `*`/`*.com`). `proxy/rules.py`, `cli/credproxy_cli/core/rules.py` + `core/pathmatch.py` (path globs: `*` within a segment, `**` across; mirrored, wire-parity tested), and `docs/reference/rules.md`.

**Structured log stream**: **every** proxy-authored line is a `credproxy {json}` record — one JSON object per line with a `kind` and an RFC-3339 `ts` (`proxy/log.py`; `audit.py` is a thin alias for `kind:"audit"` events). Debug kinds (`http`, `api`, `sni`, `scheme`, `script`, `rule-error`, `sigv4`, `main`) and audit kinds (`kind:"audit"` with an `event` subfield: credential injection, re-seal mint, and every rule hit — visible or hidden). This is **safe by construction**: an untrusted value (a rule/scheme error message that can echo workspace input) is always a JSON-encoded value, so its newlines are escaped and it can never spill a forged `credproxy {…}` record onto its own line — the substring-forgery a raw text stream allowed (`fail('\n[audit] {..}')`) is structurally impossible. mitmproxy's own termlog is left untouched. `docker logs` is the durable store (survives stop/start); `credproxy workspace NAME logs` reformats the records to one human line each (mitmproxy lines pass through verbatim), `--json` streams the raw records as JSON-lines (a non-proxy line wraps as `{"kind":"raw","line":…}`), and `--audit` filters to `kind:"audit"`. Audit events carry only structural facts (binding/rule names, host, method, query-stripped path, a coarse outcome) — **never** a secret value or a header value.

**Placeholder materialization**: missing `name` and `placeholder` fields are generated (auto-name rule: `<injector>-<provider>[-N]`; placeholder from the injector's pattern) and written back into the TOML file via surgical text edits that preserve comments. The TOML file is the single source of truth; no in-memory state diverges from disk.

## Threat model

- **Workspace container**: cannot read the host filesystem, so cannot read `$XDG_STATE_HOME/credproxy/workspaces/<name>/auth.token`. Can hit `/admin/*` endpoints over the shared netns and gets 401 without the token. No window in which `/admin/config` is unauthenticated.
- **Browser on host**: blocked by Chrome's Private Network Access (we never set `Access-Control-Allow-Private-Network`) plus the `fetch_metadata_guard` middleware (rejects requests with `Sec-Fetch-Site: cross-site`/`same-site`). Both layers act before any handler runs.
- **Other host users on a multi-user host**: can read `auth.token` (mode 0644 so the in-container uid 31337 can read it through the bind mount) and forge admin requests. Damage ceiling is DoS-or-config-replace — the user's secrets live in op://, keychain, etc., and only enter the proxy through bearer-authenticated config pushes. Documented limitation; credproxy is a single-user dev workstation tool.
- **Same-user malicious process**: out of scope (already has access to ssh keys, env vars, etc.).

## Architecture decisions that should not be casually reversed

These still apply — worth surfacing because they will tempt reconsideration:

- **Two-container shape is forced**, not chosen — netfilter must run in the same kernel as the traffic, and on macOS/Windows that kernel is inside Docker Desktop's VM. A host process cannot install iptables there. Don't propose collapsing to a single host process.
- **Transparent capture of all TCP**, not port-based selection. The product promise is "every tool works"; selective capture leaks edge cases.
- **SNI-based intercept decision**, not IP-based. CDN IP reuse breaks IP rules.
- **HTTP/3 dropped at netfilter** to force TCP fallback, not intercepted. mitmproxy QUIC is experimental.
- **IPv6 dropped entirely.**
- **Bootstrap over plain HTTP from inside the netns is fine** — no eavesdropper exists on shared loopback/link-local. This resolves the chicken-and-egg of trusting the trust source. Don't add TLS or auth to the bootstrap routes.
- **Single HTTP listener for admin + bootstrap.** Bearer auth gates `/admin/*`; bootstrap routes are open. Browsers are kept out by PNA + Sec-Fetch-Site, not by a separate listener or a separate iptables rule. Don't re-split.
- **Host-owned bearer, bind-mounted into the proxy.** The host-side `auth.token` (per-workspace, under `$XDG_STATE_HOME/credproxy/workspaces/<name>/`) is the source of truth; the proxy reads it directly from the bind mount on every admin request — no staging copy, no in-memory snapshot, rotation works without restart. Don't reintroduce TOFU or in-container token generation.
- **Providers are host-side executables, protocol is parent-agnostic.** A provider is a host executable speaking a tiny stdin/stdout JSON protocol (versioned, `op`-keyed). Today the CLI execs it; a future daemon could. Don't embed secret fetching inside the proxy or couple the protocol to the CLI. See `docs/reference/providers.md`.
- **Push model: the CLI resolves secrets and pushes resolved values to the proxy.** The proxy never calls a provider; the host CLI fetches each secret at `start`/`apply`/`binding test`/`push` and POSTs resolved literal values to `/admin/config`. Don't move provider invocation into the proxy.
- **Admin URLs are loopback-only (#41 I8).** The push wire carries **resolved secret values over plain HTTP** (there is no TLS on the admin API), so every admin URL — an `attach.admin_url`, `credproxy push --admin`, a discovered container's published port — must resolve to `127.0.0.0/8`/`localhost` (the same host, via a published ephemeral port or the shared netns). `core/push.require_loopback` is the one enforcement point (config-load and dispatch). Don't relax it or add TLS to make a remote proxy "work"; a remote proxy is out of the model.
- **Attach registration is host-side only (#41 I9).** Container-supplied data (labels, env, names) must never *select* a host-side secret source: the host-declared `attach` selector matches containers, never the reverse. This forecloses the wrong daemon design where a container label names its own binding set — a rogue container wearing a crafted label would get real secrets pushed to itself. Any future `push --watch`/daemon matches **discovered containers against host-registered workspaces**, full stop.
- **Placeholder materialization written back to the config file.** Generated names and placeholders are inserted as surgical TOML edits (preserving comments and ordering); the file is the single source of truth. Don't hold generated values in memory or a separate state file.
- **Ephemeral proxy host port, resolved at call time.** The proxy container is launched with `-p 127.0.0.1:0:39998` (Docker assigns a free port); the CLI resolves it via `docker port` at each call rather than persisting it. This is what allows multiple workspaces to run concurrently without port conflicts. Don't hardcode the host port or cache it.
- **Strict-by-default two-surface CLI.** `credproxy` is strict (explicit names, no defaults, no prompts, scriptable); `credp` is the human alias (`credproxy --loose`). Convenience lives in the alias, not the default binary behavior, so scripts that omit the flag fail safe. Don't flip the default.
- **cwd-addressing is a loose-only resolver; the name stays canonical.** A workspace's optional `directory` field lets `credp` resolve an omitted NAME by current directory (walk-up, longest-prefix, cwd before the default pointer, announced). It's a resolver layered on top — never a rename of the addressing model. Don't let the strict surface consult cwd, don't move config/state into the project dir (state stays central under `$XDG_STATE_HOME`, keyed by name), and don't add a path→name index (the `directory` field in each TOML is the single source; resolution scans them tolerantly via `quick_directory`). `core/dirmatch.py`, consulted only in `_resolve_ws`.
- **Proxy container holds the proxy core; host plugins (future) handle host-touchy things.** Don't push host-touchy logic into the proxy to "simplify"; it breaks cross-platform.

## Non-goals (don't accidentally implement)

- HTTP/3/QUIC interception, IPv6, DNS interception, hostname-based egress allowlisting, process attribution (PID), cert-pinning workarounds, mTLS injection, multi-workspace-per-proxy, bypass-resistance against an adversarial workspace. credproxy is a developer convenience boundary, not a hardened jail. (The **rules** layer sits adjacent to egress-allowlisting but does not cross the line: rules govern HTTP on hosts you *name*; unnamed hosts stay SNI-passthrough, untouched and unblockable, and there is no deny-by-default — hostmatch structurally rejects `*`/`*.com`, so "block everything except X" is inexpressible.)
- Multi-user host support: documented limitation, not a feature.
- Auto-env-var push convenience (the proxy pushing conventional env vars into the workspace environment directly): deferred. The pull model via `/setup` works; the push layer is a future layering on top.

## Key constants

All declared as `ENV` in `proxy/Dockerfile` — single source of truth for both the in-container code (via the inherited process environment) and the host CLI (via `docker inspect`'s `Config.Env`). Bumping any value is an image/CLI API break: rebuild after with `credproxy dev build` (now detected — the stale-image warning on `start`/`doctor` fires when the built image's `credproxy.src_digest` label no longer matches the checkout).

In-container runtime values (read by `entrypoint.sh` and `constants.py`):

- `CREDPROXY_MITMPROXY_UID=31337` — mitmproxy runs as this uid; `useradd` in the Dockerfile and the iptables `-m owner --uid-owner` rule depend on it (prevents redirect loop on mitmproxy's own outbound).
- `CREDPROXY_PROXY_PORT=39999` — mitmproxy transparent-intercept bind port. Picked unusual to minimize collision with workspace-side dev tools.
- `CREDPROXY_SENTINEL_IP=169.254.1.1` — link-local for the workspace-facing endpoint, resolved as `proxy.local` from the workspace side. iptables redirects `<sentinel>:80` to `CREDPROXY_HTTP_PORT`.

CLI/image contract (also read by `bin/credproxy` from `Config.Env`):

- `CREDPROXY_HTTP_PORT=39998` — merged HTTP API bind port (admin + bootstrap). CLI publishes it as an ephemeral host port (`-p 127.0.0.1:0:39998`) and resolves the actual port via `docker port` at call time.
- `CREDPROXY_TMPFS=/run/secrets` — CLI sets up `--tmpfs` here with `mode=1777` (so the proxy's unprivileged uid can write `config.json`, including after a `docker start`); `admin.py` writes `config.json` to this directory.
- `CREDPROXY_TOKEN_PATH=/run/secrets-ro/auth.token` — CLI bind-mounts the host token here; `admin.py` reads from this exact path.
- `CREDPROXY_SOURCE=/opt/proxy` — CLI bind-mounts `proxy/` here for dev (live edits + `credproxy dev reload`); also the Dockerfile `WORKDIR`.

## Storage layout

XDG-based. Override with the XDG env vars (works for tests too).

```
$XDG_CONFIG_HOME/credproxy/                      # default ~/.config/credproxy/ (tier 1: user)
  workspaces/<name>.toml                          # per-workspace config
  workspace.template.toml                         # optional personal scaffold override (shadows all)
  injectors/<name>.toml                           # user injector registry
  providers/<name>  (or <name>/run)               # user provider registry
  scripts/<name>.star                             # user script registry
  presets/<name>.toml                             # user preset registry

<repo>/overlay/<org>/  (or $CREDPROXY_OVERLAY_PATH)   # org overlays (tier 2, N ordered): each
  workspace.template.toml                         #   SUBDIR of the overlay/ container is one
  injectors/ providers/ scripts/ presets/         #   overlay; container upstream-empty
                                                  #   (scaffold + registry overrides/additions)

cli/credproxy_cli/builtin/                        # upstream defaults (tier 3)
  workspace.template.toml                         # the scaffold `create` stamps
  workspace.attach.template.toml                  # the scaffold `create --attach` stamps
  injectors/ providers/ scripts/ presets/         # builtin definitions

$XDG_STATE_HOME/credproxy/                        # default ~/.local/state/credproxy/
  default-workspace                               # current-default pointer (loose surface)
  locks/<sha256>.lock                             # per-target push locks (stateless `credproxy push`)
  workspaces/<name>/
    auth.token                                    # bearer token (mode 0644)
    setup_done                                    # container id that last completed setup
    applied-spec.json                             # last applied container spec (for drift)
    applied-bindings.json                         # last pushed bindings metadata (no secrets)
    applied-rules.json                            # last pushed rules metadata (no secrets)
    push.lock                                      # per-workspace push lock (wait-then-repush)
    sessions/<pid>                                # pidfiles for auto-stop tracking
```

**Layered overlay resolution (the fork/customization seam).** Every customizable asset resolves through one ordered search path, most specific first: **user** (`$XDG_CONFIG_HOME/credproxy/`, per-machine) → **overlays** (org tier — `$CREDPROXY_OVERLAY_PATH`, or by default the subdirs of the `<repo>/overlay/` container) → **builtin** (upstream defaults in `cli/credproxy_cli/builtin/`). The middle tier is **N ordered overlays, not one**: `CREDPROXY_OVERLAY_PATH` is an `os.pathsep`-separated list searched leftmost-first (PATH semantics), replacing the default entirely — **unset** → **discovery**: each SUBDIRECTORY of the `<repo>/overlay/` container is one overlay, lexical order by basename (the container ships upstream-empty except a README, so a fork activates an overlay with just `mkdir overlay/<org>/`; files in the container are skipped; use numeric prefixes `10-base/`, `20-team/` when stacking order matters — and don't park scratch dirs there, any subdir activates); **set-but-empty** (`""`) → NO overlays (explicit opt-out); empty entries within the list are skipped. Overlays are labelled `overlay:<dir basename>`, with a deterministic numeric suffix (`overlay:base`, `overlay:base#2`) for duplicate basenames; these labels flow through `Injector.source`/`Provider.source`/`Script.source_origin`, `info`, `list`, and doctor check ids (derived only from `overlay_dirs()`, so a given env var always yields the same labels). A same-named file in a higher tier shadows the lower; a new name adds. The single seam is `paths.overlay_roots()` (`user > overlays > builtin`), which both `paths.layered_dirs()` (registries) and `paths.resolve_singleton()` (the singletons `workspace.template.toml` and `workspace.attach.template.toml`) derive from — so a singleton rides the **same** walk and a user's personal `$XDG_CONFIG_HOME/credproxy/workspace.template.toml` shadows every overlay's. The registries treat the tier label as opaque, so N overlays need zero per-registry logic. The template is a **literal** workspace config the scaffold stamps: every occurrence of the exact token `{name}` is replaced (plain `str.replace`, **no** `str.format`, so literal braces need no doubling). The default *workspace* image is therefore just the `image` line in that template (no separate knob); `image` is mandatory in `load_config` (no built-in default — the scaffold always writes one). The *proxy* image tag is a plain engine constant in `paths.py` (`IMAGE_TAG`), deliberately **not** customizable for now; `home` has no fallback — it is optional sugar for a managed volume (omit for an ephemeral home). Overlays let an org customize the scaffold + definitions **without editing engine code or maintaining a conflicting fork** — upstream ships `overlay/` empty, so a fork only ever adds files there (its whole diff lives in `overlay/`) and `CREDPROXY_OVERLAY_PATH` gives a no-fork path (and lets you stack a team overlay over an org-base one). See `docs/advanced/overlays.md`.

## Commands

Two entry points:

- `bin/credproxy` — **strict surface**: explicit workspace names everywhere, no default resolution, no prompts, no aliases, scriptable. The contract.
- `bin/credp` — **loose surface** (`credproxy --loose`): adds default-workspace resolution (announced on stderr), short aliases, and a confirmation gate on destructive-and-implicit actions. The human entry point.

`--json` is orthogonal: available on both surfaces; errors serialize as JSON, streaming commands (`logs`) emit JSON-lines.

**Workspace commands (canonical):**

- `credproxy workspace create NAME [--here | --dir PATH] [--attach SELECTOR]` — scaffold `<name>.toml` + `auth.token`. Does not start anything. The scaffold sets a concrete `image` (the builtin/overlay `workspace.template.toml`); to use a different image, edit the generated file. `--attach SELECTOR` (`compose-project=P` | `container=X` | `admin-url=URL` | `discover=k=v[,k=v]`) instead scaffolds an **attached** workspace from `workspace.attach.template.toml` (rides `resolve_singleton()`, overlay/user-overridable) — no container fields; use `push`, not `start` (the follow-up hint points there). `--here`/`--dir` records a `directory` association (cwd-addressing; see below). On the **loose** surface NAME may be omitted with `--here`/`--dir` — it is derived from the directory's basename (sanitized to the name charset, deduped with a numeric suffix vs existing/reserved names); strict always requires NAME. On the loose surface only, if no default workspace is set yet, the new one becomes the default (announced); it never overrides an existing default.
- `credproxy workspace use NAME` — set the default workspace pointer (loose surface only).
- `credproxy workspace NAME bind-dir [--dir PATH]` — associate the workspace with a host `directory` (default: cwd), so the loose surface resolves it by cwd. Surgical edit of the `directory` field; the TOML stays the source of truth. See **cwd-addressing** below.
- `credp current` — **loose surface only** (the read companion to the loose-only `use`; both sides of the default-workspace concept live on loose, and the strict surface has no implicit target). Reports the workspace a bare verb targets, *distinct from* the default pointer: mirrors `_resolve_ws` resolution — cwd match first, then default — printing the effective workspace name on stdout (clean for `$(credp current)`) and announcing the source, plus any shadowed default, on stderr. `--json` emits `{"workspace", "source": "directory"|"default"|null, "default"}` (the default pointer is always carried separately, never conflated with the effective target).
- `credproxy workspace list [FILTER]` (or `credproxy list`) — workspaces with running status and image. **Strict is a plain inventory** (no default/cwd markers — strict consults neither the pointer nor cwd). **Loose** adds two legended implicit-targeting markers: `*` default, `→` current directory (both can land on one row; the `→` slot and a stderr legend appear only when cwd matches a workspace, so a no-cwd survey renders unchanged). The DIRECTORY column is factual config and shows on both surfaces. `--json` carries `default` and `here` booleans per row (both always `false` on strict).
- `credproxy info` — inspect the **centralized** (non-workspace) config and state: the resolved roots (config/state/builtin dirs), the **ordered overlays** (label · path · present, most specific first), the proxy `IMAGE_TAG`, a **per-tier registry breakdown** (effective injector/provider/script/preset counts keyed by full tier label — `user`, each `overlay:<name>`, `builtin` — the "is my overlay active, and which override won?" signal, with an `overlay_overrides` rollup counting registry entries resolving from any `overlay:*` tier plus overlay-supplied singletons; **effective** view, so an overlay asset a user file shadows counts as `user`), and the env overrides in effect (`XDG_*`, `CREDPROXY_OVERLAY_PATH`, `VISUAL`/`EDITOR`). `--json` emits `overlays: [{label, path, present}]` (ordered) and registry counters keyed by full label. The default-workspace pointer is a loose-only concept, so it appears only on the loose surface (consistent with `list`/`current`); everything else is surface-agnostic. `--json` emits the whole structure (subsumes any standalone "read the default pointer" command — the default is one field). The complement to the per-workspace `inspect`. Reads only.
- `credproxy doctor [NAME] [--fetch]` — environment preflight + config validation in one sweep, reporting **all** failures (unlike `start`/`binding add`, which fail at the first): docker on PATH + daemon reachable, proxy image present + ENV-contract valid (plus a **passing** `image:fresh` check that compares the checkout's `proxy/` digest against the image's `credproxy.src_digest` label — a drift/no-label/skip is `ok=True` with a rebuild hint, never a failure, since the old image still works), and — only when `CREDPROXY_OVERLAY_PATH` is set — one existence check **per env-listed overlay** (index-qualified id `overlay[i]:<base>:exists`; a configured-but-missing entry FAILS; discovered `<repo>/overlay/` subdirs exist by construction, so unset env → no overlay checks; resolution elsewhere stays tolerant); then per-workspace (an explicit NAME goes through the same `for_name` validation as every other command; else every workspace) `load_config`, a two-layer binding check (report-all per-binding probes — injector/provider resolution, each host glob — **plus** the authoritative aggregate `load_bindings`, the same parse+validate `start` runs: required fields, name/secret-slot/collision, script resolution), and the symmetric `load_rules` for the `[[rule]]` layer. So a green doctor means `start`'s static validation passes — the command's whole premise. `--fetch` (opt-in; providers can prompt/unlock, so it **requires an explicit NAME** — never fanned out across all workspaces) additionally resolves every binding's secret, reusing the same parse (so `doctor NAME` and `doctor NAME --fetch` never disagree on the static verdict). Check ids are index-qualified (`ws:<name>:binding[i]:host[j]`) so a `--json` consumer keying by id can't collide two failures. Exit 0 iff all pass; `--json` emits the structured `{id, ok, message, hint}` check list (`core/doctor.py`). The action-time validators (`load_config`, `load_bindings`/`load_rules`, `find_injector`/`find_provider`, `hostmatch`, `ImageEnv.load`) reused, report-not-raise.
- `credproxy workspace NAME enter [-- CMD...]` — hot path: start if needed, `docker exec` into the workspace. With no `-- CMD` it runs the config `shell` (default a login shell, `bash -l` — entering is "logging in", the ssh model); `-- CMD` runs a bare, non-login command (the ssh `host cmd` model). Tracks auto-stop sessions. The primary verb.
- `credproxy workspace NAME exec [--login|--raw] [--user U] [--push] -- CMD...` — the **non-interactive** sibling of `enter` for scripted/agent use: start if needed, `docker exec` `CMD`, propagate its exit code. It requires a `-- CMD` (no default shell). **Environment (default = parity with `enter -- CMD`):** the default wraps `CMD` in the **same env shim** `enter` uses (`sh -c '<enter_prelude>; exec "$@"'`), so the CA-trust env is sourced and `exec -- curl https://…` works against the intercepting proxy — *not* raw argv (that silently broke TLS, the exec headline). `--raw` opts into a direct `execve` (no shell wrapper, no CA env — minimal/distroless images); `--login` a `bash -lc` login shell (profile.d + rc, mise shims); the two are mutually exclusive. `--user U` overrides config `user` for the call. **Auto-stop:** it never *initiates* one, but DOES register a session pidfile for the command's duration — so a concurrent `enter` teardown counts the in-flight exec and can't `docker stop` the box under it (visible to the reaper without becoming one). **Fast path:** both containers already running + no `--push` → skip the full `start` reconciliation (drift/config-push staleness is what `start`/`apply` are for; `--push` forces it). A signal death maps to `128+N` (SIGINT→130), so the propagated code matches the process. **`--json` doesn't apply** and is refused — `exec` is a transparent pipe (the command's stdout is arbitrary bytes, not credproxy's to structure; the exit code is the exit code). `enter`/`exec` share `_docker_exec_argv` so they can't drift on workdir/user/exec_flags/session-control ordering. `lifecycle.exec_workspace`/`_exec_cmd`/`_start_for_exec`.
- `credproxy workspace NAME start` — (re)start the proxy container, wait for `/health`, push bindings config, then (re)start the workspace container. `/health` is a **capture-readiness** probe, not liveness: it stays `503` (with a `pending` list) until the mitmproxy transparent listener accepts **and** the CA exists (the handler probes the port live rather than trusting an addon flag), so `start` never hands off the workspace during the boot window where TLS intercept isn't actually up. Creds-readiness (config pushed) is deliberately a separate future signal — folding it into `/health` would deadlock `start`, which waits on `/health` *then* pushes. Auto-creates if missing. Recreates a container when its spec drifts (proxy: image changed; workspace: image/home/mounts/env/setup or proxy netns peer changed, tracked via `credproxy.spec` label). **Proxy-image gate** (`ensure_proxy_image`, porcelain-side since it prompts — `recreate` calls it too, `exec`'s already-running fast path is exempt): a **missing** proxy image is offered as an inline build on loose+TTY (`[Y/n]`, default yes; `--yes` builds unprompted; loose-no-TTY and strict fail closed with the exact remedy `build it with: credproxy dev build`, serialized as `ImageError` under `--json`) instead of the bare `ImageEnv.load` backstop; a **present-but-stale** image (checkout's `proxy/` digest ≠ the image's `credproxy.src_digest` label) warns and proceeds — loose+TTY offers a rebuild (`[y/N]`, default **No**, so the working old image is never blocked; `--yes` takes the default No, no surprise rebuild), strict prints a one-line stderr warning, a **no-label** image (built pre-digest) is "unknown" → warn, never a rebuild prompt; skipped silently with no repo checkout.
- `credproxy workspace NAME stop` — `docker stop -t 1` both containers; kept, not removed.
- `credproxy workspace NAME recreate [--proxy] [--reset-volume NAME ...]` — force-rebuild the workspace container from a clean slate (re-runs `setup`), then start it. Unlike `delete`, preserves all managed volumes, config, auth token, and state — only the container is replaced. Default rebuilds the **workspace container only** (keeps the running proxy + its CA, faster); `--proxy` (alias `--all`) also recreates the proxy, regenerating its CA (full re-bootstrap; the workspace re-bootstraps CA trust). `--reset-volume NAME` (repeatable) *also* wipes that managed volume (e.g. `--reset-volume home`, re-seeded from the image) while keeping config/token/state — bind/overlay host-path mounts are untouched. It's the one recreate mode that destroys data, so it's gated like `delete` (confirm on an implicit default in loose mode; `--yes` bypasses). Implemented as `docker rm -f` the target(s) [+ `docker volume rm` the named volume(s)] + `start` (so it reuses the health-wait, config push, and setup re-run).
- `credproxy workspace NAME delete [--keep-volumes]` — remove both containers, the workspace's managed volumes (by name prefix), config file, and state dir; `--keep-volumes` preserves the volumes. In loose mode, prompts when targeting the default workspace.
- `credproxy workspace NAME apply` — best-effort reconcile: bindings drift → live re-push to the running proxy; container-spec drift (image/home/mounts/env/setup) → deferred with a restart hint. Reports what was applied vs. deferred. On an **attached** workspace `apply` ≡ `push` (no container spec to reconcile).
- `credproxy workspace NAME push [--wait] [--timeout SECS]` — resolve every binding's secret and POST the full wire config (**bindings + rules**) to the workspace's proxy: a managed workspace's published port (start it first — a stopped proxy errors toward `start`, not auto-start), or an attached workspace's resolved `attach` target. The extracted resolve+POST engine (`core/push.py`) that `start`/`apply` also call — atomic/fail-closed, blocking `flock` (wait-then-repush, never skip). `--wait` polls `/health` (never `/ready`, I1) until capture-ready first. Loose alias `credp push [NAME]`.
- `credproxy push --admin URL --config FILE --token FILE [--wait] [--timeout SECS]` — **stateless** top-level escape hatch (no workspace/state, both surfaces): push a `[[binding]]`/`[[rule]]`-subset FILE (container/`attach` keys rejected, same validators) to a loopback (I8) proxy, token from FILE. Lock keyed by target hash under `locks/`.
- `credproxy workspace NAME resolve (--json | --out FILE)` — build the full wire config **with resolved secret values** without contacting any proxy (the deliver-at-creation channel). `--json` to stdout (host-transient); `--out FILE` writes mode 0600 and warns if outside the state dir (it holds real secrets). Feeds the stateless `push --config`.
- `credproxy emit-compose [NAME] [--image TAG]` — top-level, the one deliberately Compose-aware command: print a Docker Compose fragment (proxy service — NET_ADMIN, mode-1777 tmpfs, read-only token bind, ephemeral loopback admin port, healthcheck on **`/ready`** via a `python -c` urllib probe since the image has no curl — plus the workspace service's `network_mode: "service:proxy"` + `depends_on: service_healthy`). Every port/path from the image's `ENV` contract via `ImageEnv.load` (no hardcoded constants). NAME bakes that workspace's real token path; without NAME, `${CREDPROXY_STATE:?…}/auth.token` for a Compose `.env`. `--json` refused (it emits YAML).
- `credproxy workspace NAME inspect` — config + running state + host port + binding summary + rule summary + itemized drift against applied-spec/applied-bindings/applied-rules; for an attached workspace, the `attach` selector + resolved target instead of container state.
- `credproxy workspace NAME config [--declared]` — dump the container-side config. Default `effective` (every field with its in-effect value, all defaults filled — including the enter-time `workdir`→home and `enter_prelude`→shim defaults that `inspect` leaves null); `--declared` shows only what's literally in the TOML. `--json` on both.
- `credproxy workspace NAME edit` — open `<name>.toml` in `$VISUAL`/`$EDITOR` (default `vi`), then validate it; warns if the edit left it invalid, else hints `apply`/`start`. Sugar over editing the file directly; interactive, so no `--json`.
- `credproxy workspace NAME logs [--audit] [--json]` — tail the proxy container's structured log stream (`docker logs -f`), reformatting each `credproxy {json}` record to a human line (mitmproxy's own output passes through verbatim). `--json` streams the raw records; `--audit` filters to `kind:"audit"` governance events.

**Binding commands (through a workspace):**

- `credproxy workspace NAME binding add --injector INJ --provider PROV --secret REF --host HOST [--host HOST...] [--name NAME] [--placeholder PH] [--env VAR]` — `--secret` repeats as `SLOT=REF` for a multi-slot secret (bare `REF` = single-slot). Single-binding path only; coordinated sets are the `preset` noun (below), NOT a `--preset` flag here (retired).
- `credproxy workspace NAME binding remove BINDING_NAME`
- `credproxy workspace NAME binding list`
- `credproxy workspace NAME binding test [BINDING_NAME]` — dry-run fetch via provider (reports secret length, not value); exit 1 if any fail.
- `credproxy workspace binding test --provider PROV --secret REF [--injector INJ]` — ad-hoc mode: test a definition *before* it is bound. No workspace is resolved or required (the workspace name is ignored); validates that the provider/injector resolve and the fetch succeeds.

**Rule commands (through a workspace):** the credential-free traffic-governance layer (see the Rules note above; `docs/reference/rules.md`). Mirrors the `binding` noun.

- `credproxy workspace NAME rule add {block|respond|rewrite|script} --host HOST [--host HOST...] [--method M...] [--path GLOB] [--name NAME] [--visible|--hidden]` — the action is an argparse **subcommand** (not a `--action` flag), so each action owns exactly its params and a misplaced flag is rejected structurally: `block` `[--status N]`; `respond` `--status N [--body TEXT] [--header K=V...]`; `rewrite` `[--header K=V] [--remove-header NAME] [--resp-header K=V] [--resp-remove-header NAME]`; `script` `--script NAME`. Auto-names/materializes like bindings; the CLI builds the `[[rule]]` entry and routes it through the single field validator `core/rules._parse_rule_entry` (shared with the load path, mirrored to the proxy's `_load_rules`), then `validate` for cross-rule/semantic checks (unique names, host/path glob, Host/:authority ban, script resolution).
- `credproxy workspace NAME rule remove RULE_NAME` — gated like `binding remove` on an implicit default (loose).
- `credproxy workspace NAME rule list` — marks hidden rules `HIDDEN`.
- `credproxy workspace NAME rule test METHOD URL [--live]` — dry-run evaluator (the "why was/wasn't this blocked?" tool): reports the matched rule + resolved action. **Default (offline):** host-side matcher against the config file, no running proxy needed; shares the matcher with the proxy (wire-parity tested) but can't run Starlark, so a `script` rule's phase is unknown and reported conservatively (always possibly-terminal, later rules conditional). **`--live`:** asks the running proxy (`POST /admin/rule-test`, bearer-gated, read-only) to evaluate against its **loaded** config with its own matcher + compiled scripts — exact per-script phase (`RuleSet.dry_run`) + the intercept decision; reflects what's actually running (may lag the edited TOML until `apply`/`start`).

**Preset commands:** a *preset* is a **service setup pack** — the coordinated bindings a credential needs across a service's hosts AND the credential-free rule guardrails that should accompany them, in one `<name>.toml` in the layered registry. Promoted from the old "one credential's binding shape" (`binding add --preset`, **retired**) to a first-class noun.

- `credproxy workspace NAME preset add PRESET [--provider PROV] [--secret REF]` — the single entry point. Expands the preset's `[[part]]` bindings (sharing one placeholder) **and** its `[[rule]]` guardrails, materializes names, and stamps ordinary `[[binding]]`/`[[rule]]` blocks (**expansion, not a link** — the proxy never sees a "preset"; edit/remove after is normal). Both halves are optional: a **binding-only** preset (needs `--provider`/`--secret`, or the preset's defaults — `github` defaults `gh-cli`/`github.com`), or a **pure-rule policy pack** (no `[placeholder]`/provider/secret; `preset add org-guardrails` with zero flags). Rules carry a `suffix` (like parts) → `name = "<preset>-<suffix>"`, are validated via `core/rules._parse_rule_entry` (same validator + `[rule.params]` support), and `[rule.params]` sub-tables are stamped in. **Atomic:** a generated `<preset>-<suffix>` colliding with an existing binding/rule name fails the whole add before any write. **Announces** bindings written, rules written, and **any host newly TLS-intercepted** (a preset rule on a bindings-free host flips it via the UNION intercept set — a CA-cert-error tripwire otherwise). `preset` is dual-role: `workspace NAME preset add` (workspace-scoped) + top-level `preset list` (below).
- `credproxy preset list` — top-level definition command (sibling of `injector list`/`provider list`); enumerates each preset's expansion (bindings **and** rules) so the operator sees the full stamp before applying (`describe_presets`).

**Mount commands (through a workspace):**

- `credproxy workspace NAME mount add --volume VOL --target PATH [--ro] [--preserve] [--user-owned]` — add a managed-volume mount (a named, persistent, per-workspace Docker volume) via a surgical TOML edit. `--volume home` writes the `home = "..."` sugar instead of a `mounts` entry. `--user-owned` (writes `user_owned = true`) chowns the volume to the workspace `user` so a non-root user can write a volume mounted at an image-absent path (which Docker would otherwise create root-owned); requires a non-root `user`, rejected for the `home` sugar (use an explicit `[[mounts]]` table with `volume = "home"`). See the mounts model below for the chown semantics. Without `--preserve` it is a deferred config edit (the volume is image-seeded on the next `start`, like any added volume). With `--preserve` it captures the **current** container's data at `PATH` into the new volume (`docker cp ... - | tar -x` into a helper that mounts the volume; the helper inherits the workspace's userns mapping + runs as namespace-root so ownership round-trips on rootless podman) and then recreates the container so the volume mounts populated — a non-empty volume suppresses Docker's image-seed, preserving the accumulated writable-layer data that a plain recreate would lose. A mount can't attach to a live container, so any new mount forces a recreate; `--preserve` just carries the data across it. On a **running** workspace with **live `enter` sessions** (which the recreate kills) it is gated: strict refuses without `--yes`; loose prompts (fails closed without a TTY); a running-but-idle workspace is not gated (parity with un-gated plain `recreate`). Only managed volumes (incl. `home`); bind/overlay mounts stay hand-edited.

**Definition commands:**

- `credproxy injector scaffold NAME` — copy the builtin `bearer` template to `$XDG_CONFIG_HOME/credproxy/injectors/<name>.toml`.
- `credproxy injector list`
- `credproxy provider scaffold NAME` — copy the builtin `env` template to `$XDG_CONFIG_HOME/credproxy/providers/<name>`.
- `credproxy provider list`
- `credproxy script check [NAME]` — compile resolvable `.star` scripts in the proxy runtime **before** push (a syntax error otherwise surfaces at `start`), with the same on-host-or-image fallback as `dev test` (on-host needs only the `starlark` dep — no daemon, no `CREDPROXY_*` env). Classification is by reference, not filename: a script named by a resolvable `scheme="script"` injector manifest compiles under the INJECTOR profile **paired with that manifest** (so a slot/family mistake surfaces too); an unreferenced script is tried under BOTH the injector and rule profiles and passes if either compiles, reporting which. Exit 0 iff all compile; `--json` emits per-script `{name, origin, ok, error, profiles}`. `doctor NAME` reuses the same on-host compile for a workspace's scripted injectors (skip-with-note when the runtime isn't importable, so doctor stays daemon-free). `core/scriptcheck.py`.

**Loose aliases** (loose surface / `credp` only):

`credp enter [NAME]`, `credp exec [NAME] [--login|--raw] [--user U] [--push] -- CMD...`, `credp start [NAME]`, `credp stop [NAME]`, `credp recreate [NAME] [--proxy] [--reset-volume NAME]`, `credp delete [NAME] [--keep-volumes]`, `credp apply [NAME]`, `credp inspect [NAME]`, `credp config [NAME] [--declared]`, `credp logs [NAME]`, `credp bind-dir [NAME] [--dir PATH]`, `credp use NAME`, `credp create [NAME] [--here]`, `credp list [FILTER]`, `credp binding {add|remove|list|test} ...`, `credp rule {add|remove|list|test} ...`, `credp preset add PRESET ...` — all resolve to the canonical workspace command with the **implicit workspace** as subject. An explicit NAME overrides resolution. (`credp preset list` stays top-level — it's definitional, not workspace-scoped.)

**cwd-addressing (loose surface).** When NAME is omitted, the loose surface resolves the implicit workspace by **current directory first, then the default pointer**: a workspace whose `directory` field is an ancestor of (or equals) cwd matches, most-specific (longest-prefix) wins, and the choice is announced on stderr (`(matched current directory)` vs `(default)`). It's the `cd project && credp enter` ergonomic. The name stays canonical — `directory` is just another resolver, a sibling of the default pointer (`core/dirmatch.py`, consulted only in `_resolve_ws`). Strict `credproxy` never consults cwd. Ambiguity (two workspaces, same dir) errors; `/` and `$HOME` are ignored as too broad. The field is host-side resolution metadata only — never in the spec hash, no recreate. Set via `create --here`/`--dir` or `bind-dir`. See `docs/reference/configuration.md`.

**Harness commands** — `credproxy dev ...`, for hacking on credproxy itself; need the repo checkout:

- `credproxy dev build` — `docker build` the proxy image, stamping `--label credproxy.src_digest=<digest>` where `<digest>` is `paths.proxy_src_digest()` — a deterministic sha256 over the sorted `proxy/` tree (paths + contents; `__pycache__`/`*.pyc` ignored, no git dependency). `start`/`doctor` recompute it to detect a stale image.
- `credproxy dev test [--cli|--proxy] [--container] [-- PYTEST_ARGS...]` — runs the host CLI suite (`tests/cli/`) and the proxy suite (`tests/`) — the CLI suite includes `tests/cli/test_docs_lint.py`, a docs-staleness guard that validates every `credproxy`/`credp` command line in README + `docs/**` against the real argparse parser tree and every relative markdown link/anchor against the target headings. The proxy suite runs **on-host** when its runtime deps (`mitmproxy`, `aiohttp`, `starlark`) import there (near-instant, skips container startup), falling back to `docker run` in the image otherwise; `--container`/`--docker` forces the image (the canonical, version-pinned env). Trailing args after `--` pass through to the proxy pytest. **Overlay tests**: every configured overlay (`CREDPROXY_OVERLAY_PATH`) with a `tests/` subdir runs as its **own** pytest invocation (never merged into the repo suite — the `test_*.py` basenames would collide under one rootdir, the same reason `tests/cli/` is split out), same on-host-or-image fallback, combined exit code. In the image the **whole overlay chain** is bind-mounted read-only at `/opt/overlays/<i>` and `CREDPROXY_OVERLAY_PATH` is rewritten to those container paths (declared order preserved), so resolution inside the container sees the overlays and every tier is present even when only one overlay's tests run; on-host the overlay run gets `PYTHONPATH` covering `proxy/` (`import testkit`) + `cli/`. Overlay authors write these with the **testkit** (see below). Working in an environment **without a docker daemon** (Claude Code on the web, CI sandboxes): see `docs/dev-environment.md` for the venv setup, the proxy suite's env-var contract, and a known environmental test failure.

  **Testkit (`proxy/testkit.py`)** — the supported harness for unit-testing scripts, importable both on-host and in the proxy image (it lazily adds `<this>/../../cli` to `sys.path` so `credproxy_cli.core` resolves at `repo/cli` on-host and the `/opt/cli` `dev test` mount in-image; **nothing in the proxy runtime imports it**). It builds each scheme through the SAME manifest+script pairing the wire/push loader uses (`find_injector`+`find_script`; `schemes.SCHEMES` for a built-in), so a manifest that disagrees with its script fails the test rather than a hand-rolled `ScriptedScheme(...)` passing against a config the proxy would reject. API: `load_injector(name).run(req, secrets, params=None, placeholder=None)→.injected` (slot-set validated against the manifest, mirroring `load_resolved`); `make_request(method, url, headers=None, body=b"")` (strips treq's default headers + bogus `content-length`, sets `Host` from the URL — the two footguns every test open-coded); `load_rule_script(name)`/`run_rule(script, req, params=None)→` a block/respond/rewrite outcome. The builtin `ovh`/`jwt-bearer` script tests are migrated onto it (dogfood). No `api` versioning yet — tracks the current script `api`, revisit when it first bumps. See `docs/advanced/overlays.md` "Testing your overlay".
- `credproxy dev reload [NAME]` — SIGHUP the workspace's proxy; python re-execs in place, picking up edited source from the bind-mounted `proxy/`. The container, netns, iptables rules, and tmpfs all survive, so pushed config persists. A python crash takes the container down (no supervisor); recover with `credproxy workspace NAME start`.

**Safety gate** (loose surface only): destructive commands that operate on an **implicit** (defaulted) workspace require confirmation. Explicit targets never prompt. `--yes` / `-y` bypasses. Fails closed without a TTY. Destructive set: `delete`, `binding remove`, `rule remove` (all high recovery cost), and `recreate --reset-volume` (wipes a managed volume's data). `stop` is not gated (restart is cheap), and neither is a plain `recreate` (all managed volumes, config, token, and state survive — only the rebuildable container is lost); the gate is flag-conditional for `recreate`, firing only with `--reset-volume`.

## Open design questions

Surface these rather than picking silently if your work touches one:

- **Injector authoring contract (resolved, fully implemented).** Injection is a typed, scheme-aware request transform: an injector selects a `scheme` (+ `params`) the proxy implements, replacing the original `format`/substring-replace. The `format` field is gone. All of it now ships: substitute + sign (`sigv4`, plus `ovh`/`jwt-bearer` as scripts) + re-seal (`oauth2-reseal` + `mint` primitives, TTL'd runtime layer), and the Starlark escape hatch on the flat/implicit-ctx "option B" primitive API (`api`-versioned manifests). See `docs/reference/injectors.md`; don't reintroduce `format` or a single-mechanism substitution, and don't revert the scripted API to explicit ctx-threading.
- **Per-request vs. per-host injection.** Currently strictly per-host; no path/method matching. Host *scoping* does support glob patterns (`*.amazonaws.com`) as of the hostmatch work, but matching is still hostname-only — no path, method, or query matching, and a pattern can't be enumerated to a fixed SNI set (so the intercept decision is the `creds.intercepts(sni)` predicate, not membership on `intercept_hosts()`).
- **Proxy CA persistence across recreate.** mitmproxy's CA is regenerated on a proxy *recreate* (not on stop/start or a crash — those keep the same container). Currently a re-bootstrap cost, not silent breakage: a proxy recreate is coupled to a workspace recreate, and the workspace's CA trust lives only in its writable layer, so the workspace re-bootstraps anyway. Would become a real gap only if CA trust were made durable across workspace recreates.
- **`/llms.txt` format.** Currently free-form prose; structured/AGENTS.md-style alternatives haven't been evaluated.
- **Mounts are a typed model.** `mounts` entries are host **binds** (`"SRC:DST[:ro]"` strings), managed **volumes** (`{volume=NAME, target}` tables — namespaced per workspace, image-seeded, ownership-clean), or **overlay** mounts (`{overlay=REL, target}` — a path under an overlay dir, searched in declared order across `overlay_dirs()` and confined within the winning overlay, read-only by default, for shipping static files with a fork; reordering overlays changes which absolute path wins → spec-hash change → recreate, an intended consequence). `home = "/path"` is **sugar** for a managed volume named `home`; omit it for an ephemeral home (the deferred "optional persistent home", now delivered). The chown of runtime-fabricated mount-parents is host-bind-aware (safe under any managed volume, never under a host bind), not anchored on home. A managed volume may also opt into **`user_owned = true`** (volume-only flag): credproxy chowns the whole volume to the workspace `user` so a non-root user can write a volume mounted at a path the image doesn't populate (Docker creates such a volume root-owned — no image content to seed ownership from). It's **opt-in per volume** (never blanket — a `chown -R` would clobber a volume image-seeded with intentional ownership), runs **after `setup`** (so a setup-provisioned `user` exists; chown resolves the name in-container, sidestepping host/userns uid math), is independent of `map_host_user` (the gap exists on plain Docker too), and rides the spec hash so toggling it recreates (re-running the chown). Distinct from `chown_mount_parents` (numeric uid, pre-setup, intermediate dirs only) — `chown_user_owned_volumes` is by-name, post-setup, whole-volume. Volume lifecycle: created on first `start`, survive `recreate`/`stop`, removed on `delete` (unless `--keep-volumes`), individually wiped by `recreate --reset-volume NAME`. Add one via `mount add --volume VOL --target PATH` (surgical TOML edit); `--preserve` seeds it from the current container's data at `PATH` (capture-then-recreate) so converting an unmounted path into a volume doesn't lose the accumulated data — the one path that pre-seeds a volume from a source other than the image (so it relies on the "non-empty volume skips Docker's image-seed" rule; ownership round-trips via a userns-matched, namespace-root tar helper).
- **Distribution constants not customizable (by choice, for now).** The proxy image tag (`IMAGE_TAG`) is the one hardcoded engine constant; the default *workspace* image lives only in `workspace.template.toml` (and `home` is optional sugar — no fallback). An overlay can customize the scaffold + definitions but not `IMAGE_TAG`. If per-distribution proxy-image rebranding is wanted later, the clean path is a `proxy_image` workspace field (`ImageEnv.load(image)` already takes one) rather than reviving an engine-wide config file.
- **Build a workspace image from a Dockerfile (idea, parked — revisit when a fat-`setup` pain or an org asks).** Let a workspace declare a `build` (Dockerfile + context, mutually exclusive with `image`); `start` runs `docker build` and feeds the resulting image ID into the existing recreate path (build-on-start, recreate on image-ID change — docker's layer cache makes repeat builds cheap). **Does not break "BYO image, never modified"**: credproxy builds the user's instructions verbatim, injects nothing, runs it unprivileged — a declarative image, not a new trust model. The **netns boundary partitions the work cleanly**: the build runs *before* the container joins the proxy netns, so build egress is un-proxied and `proxy.local`/the CA aren't reachable — therefore **Dockerfile = static/public/un-credentialed provisioning; `setup` = run-time/credentialed (the proxy injects creds) + the CA bootstrap (only reachable at run-time)**. So **refuse build-time secrets** (`--build-arg`/BuildKit) on purpose — credentialed installs belong in `setup`. Composes with overlays (ship a `Dockerfile`, reuse the `overlay:` context confinement) and with `recreate --reset-volume home` (the home volume seeds from the image only on first create, so Dockerfile changes to the home dir need a home reset — a known sharp edge). Costs: a managed built image to lifecycle (tag `credproxy-ws-<name>`, prune on `delete`), build latency + a new failure surface, rebuild-trigger fuzziness (start with always-build, optimize later). Verdict: a convenience+lifecycle layer (`docker build -t … && image = …` already works by hand), low architectural risk, moderate surface (build orchestration + image lifecycle + error reporting).
